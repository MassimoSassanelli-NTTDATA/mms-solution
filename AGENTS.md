# AGENTS.md: Mobile Maintenance Platform

## Purpose

This repository is the platform wrapper for the maintenance mobile solution. It owns product-level planning and governance, not product implementation code.

## Required Agent Context

Agents must read these before implementation planning:

- `WORKSPACE.md` and `REPOSITORY_CONTEXT.md` — generated by the setup workflow, when available
- `.github/copilot-platform.json` — repository roles, dependencies, and the authoritative platform `rules`
- `.github/skills/_subrepo_*/SKILL.md` — mirrored sub-repository skills generated by `scripts/sync-subrepo-skills.ps1` (typically via Git hook)
- `architecture/solution-overview.md` — allowed dependency directions
- `docs/process/` — authoritative process definitions (issue hierarchy, development flow, task orchestration, story branching, validation, repository-context aggregation, prompt usage)
- `docs/templates/` — epic, story, task, and validation-report templates

## Source of Truth

GitHub Issues, Sub-Issues, Pull Requests and validation reports are authoritative.

## Repository Roles

Repository roles and dependencies are defined in `.github/copilot-platform.json`.

Agents must not infer responsibilities only from repository names. Use:

- `type`
- `role`
- `description`
- `dependencies`
- `agentGuidance`

## Implementation Rules

1. Implement changes only in the code repository that owns the task.
2. Treat dependency repositories primarily as read-only context unless the task explicitly belongs to that repository.
3. Search shared repositories before creating new controls, services, DTOs, sync logic, helpers, abstractions, or patterns.
4. If a story requires changes in multiple repositories, propose separate implementation tasks for the affected repositories.
5. Use the dependency graph as the authoritative source for allowed dependency directions.
6. Keep platform-level decisions and cross-repository architecture in this platform repository.

## Cross-Repository Follow-Up Format

When a required change belongs to another repository, document:

- target repository
- required change
- reason
- affected story or epic
- suggested task title
- suggested acceptance criteria
